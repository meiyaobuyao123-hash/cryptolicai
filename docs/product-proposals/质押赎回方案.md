# 质押/赎回功能 — 技术方案

## 一、DeFi 质押/赎回

### 核心链路

```
用户 → App 连接钱包 → 选择产品 → 构建交易 → 钱包签名 → 链上执行
```

### 技术实现

#### 1. 钱包连接
- **WalletConnect v2** — 支持 MetaMask、Trust Wallet、Rainbow 等 300+ 钱包
- **Coinbase Wallet SDK** — 支持 Coinbase 钱包
- Flutter 包：`walletconnect_flutter_v2` + `web3dart`
- 用户扫码或 Deep Link 连接钱包，App 获得用户地址

#### 2. 智能合约交互

| 协议 | 质押方法 | 赎回方法 | 合约接口 |
|------|---------|---------|---------|
| **Aave V3** (借贷) | `supply(asset, amount, onBehalfOf, referralCode)` | `withdraw(asset, amount, to)` | Pool 合约 |
| **Lido** (ETH质押) | `submit(referral)` + 发送 ETH | 卖出 stETH 或通过提款队列 | Lido 合约 |
| **Compound V3** | `supply(asset, amount)` | `withdraw(asset, amount)` | Comet 合约 |
| **EigenLayer** (再质押) | `depositIntoStrategy(strategy, token, amount)` | `queueWithdrawals(...)` | StrategyManager |

#### 3. 交易构建流程

```
1. 用户选择产品（如 "USDT → Aave V3, APY 5.2%"）
2. App 检查用户钱包余额是否足够
3. App 构建两笔交易：
   a. approve(AavePool, amount) — 授权 Aave 合约使用用户的 USDT
   b. supply(USDT, amount, userAddress, 0) — 存入 Aave
4. 用户在钱包中确认签名
5. App 监听交易上链确认
6. 显示质押成功，更新余额
```

#### 4. 多链支持
- 需要支持的链：Ethereum、Arbitrum、Optimism、Base、Polygon、BSC
- 每条链的合约地址不同，需要维护一个合约地址注册表
- 用户切换链时需要在钱包中切换网络

#### 5. Gas 费优化
- 预估 Gas 费并在确认前展示给用户
- 推荐 L2 链（Arbitrum/Base）上的同类产品，Gas 费低 10-100 倍
- 可接入 Gas 代付服务（Account Abstraction / ERC-4337）

### DeFi 方案的难点

| 难点 | 说明 | 解决方案 |
|------|------|---------|
| 合约集成量大 | 每个协议接口不同 | 优先集成 Top 5 协议，逐步扩展 |
| 多链复杂度 | 同一协议在不同链上地址不同 | 维护合约注册表 JSON |
| Gas 费不可控 | 以太坊主网 Gas 可能很贵 | 优先推荐 L2 |
| 安全风险 | 用户资产安全最重要 | 不碰私钥，只构建交易让钱包签名 |
| 审计合规 | 智能合约交互需安全审计 | 只调用已审计的头部协议 |

## 二、CeFi 质押/赎回

### 方案对比

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **A. 交易所 API** | 用户授权 API Key，App 代为操作 | 全自动 | 安全风险高，用户需给出 API Key |
| **B. OAuth 登录** | 用户通过交易所 OAuth 授权 | 安全，用户体验好 | 只有少数交易所支持 |
| **C. Deep Link 跳转** | 跳转到交易所 App 对应页面 | 零风险，简单 | 体验割裂 |
| **D. 聚合平台** | 接入 1inch/Paraswap 等聚合器 | 一套接口接多平台 | 只覆盖 DEX 不覆盖 CEX |

### 推荐方案：C + A 组合

#### 第一阶段：Deep Link 跳转（零风险快速上线）

```
用户看到 "Binance USDT 活期 3.5% APY"
  ↓ 点击 "去质押"
  ↓ 跳转 Binance App 的 Earn 页面（Deep Link）
  ↓ 用户在 Binance App 内完成操作
```

各交易所 Deep Link：
- Binance: `bnc://app.binance.com/en/earn`
- OKX: `okx://earn`
- Bybit: `bybit://earn`
- 如果未安装 App，跳转对应网页

#### 第二阶段：API Key 集成（高级用户）

```
用户在 App 设置中绑定交易所 API Key（只读 + 交易权限，不给提现权限）
  ↓ App 通过 API 调用交易所的 Earn 接口
  ↓ 自动完成质押/赎回
```

主要交易所 API：
- **Binance**: `POST /sapi/v1/simple-earn/flexible/subscribe`
- **OKX**: `POST /api/v5/finance/savings/purchase-redempt`
- **Bybit**: `POST /v5/earn/place-order`

### CeFi 方案的难点

| 难点 | 说明 | 解决方案 |
|------|------|---------|
| API Key 安全 | 用户的 API Key 需要安全存储 | 设备端加密存储，不上传服务器 |
| 权限最小化 | 只要交易权限，不要提现权限 | UI 引导用户创建只读+交易的子 Key |
| 各交易所接口不统一 | 每家 API 格式不同 | 抽象 Adapter 层，统一接口 |
| 合规 | 代用户操作可能涉及牌照 | 法律咨询，可能需要 MiCA/MSB 牌照 |

## 三、推荐实施路径

| 阶段 | 内容 | 周期 |
|------|------|------|
| **V1** | DeFi: WalletConnect + Aave/Lido 质押赎回 | 核心功能 |
| **V1** | CeFi: Deep Link 跳转各交易所 | 快速上线 |
| **V2** | DeFi: 扩展 Compound/EigenLayer/Pendle | 扩展覆盖 |
| **V2** | CeFi: Binance/OKX API Key 集成 | 高级功能 |
| **V3** | Account Abstraction 免 Gas 质押 | 体验优化 |
| **V3** | 一键策略组合（跨协议/跨链自动配置） | 差异化 |

## 四、安全原则

1. **永远不碰用户私钥** — 只通过 WalletConnect 发交易让用户签名
2. **API Key 设备端加密** — 不上传服务器，不存数据库
3. **只调用已审计协议** — 不接入未经审计的小协议
4. **交易前预览** — 每笔交易展示：目标合约、金额、预估 Gas、预期收益
5. **风险提示** — 每个产品标注风险等级，高风险产品需二次确认
